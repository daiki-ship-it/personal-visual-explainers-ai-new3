# Diagram Personal（パーソナル図解）

## 起動のしかた

**トリガーは「開始」だけ**（この文字列だけ、または「開始」が依頼の核になっている場合）。それ以外の入力では本スキルを起動しない。

ユーザーが「開始」したら **確認を挟まず** **Phase 0 から Phase 9 まで一気通貫**で進める。調査対象はユーザーの指定ではなく **次の固定ミッション** に従う（下記「リサーチ対象の固定」）。

図解 HTML を生成し **surge.sh にデプロイする**（Phase 9 は省略しない）。

### リサーチ対象の固定（「開始」時の SSOT）

- **範囲**: **最新のIT・AIニュース・トレンド**から、**AI講師として「知っておくべき最新」の核をちょうど3本**に絞る。
- **期間の目安**: 実行日を含む **直近1週間程度**を主戦場とし、それより前でもいま講師の仕事に効く大きな動きなら候補に含めてよい。
- **選定基準**: 本スキル「読者・仕事・ゴール」の成功定義と「典型ニーズ」に合わせ、**商品設計・集客・登壇・今後の顧問・コンサル**への影響が大きい順に優先し、ノイズになりがちな話題は捨てる。

## 読者・仕事・ゴール・「まず覚える3つ」の意味（本スキル内の正）

**読者・仕事・成功条件・ニュース3本の解釈の正はこの節だけ**とする。他ファイルでは**同一内容を別の言い回しで長く書かない**（`exemplar.md` の `diagram-maji` 比較は**対照用の短い表**に限り、各セルは箇条書きレベルに留め、**読者・成功の定義本文は繰り返さず**本節や各リファレンスへのリンクに寄せる。チェックリストは本節へのリンクで足りる）。Phase 7 のレビュー用ペルソナは**この節と一致**させる（プロンプト内に全文を繰り返さず、実行時にこの節を読む前提にする）。

### 読者

- **読者は自分ひとり**。
- AI講師として活動はしているが、**IT業界全般への苦手意識が強く、IT業界での勤務経験はない**。
- **IT専門用語はほぼわからず、抵抗感がある**。長文や用語の連続で疲れやすい。
- **ITリテラシーは低め**とみなし、誰が読んでも通じるレベルまで噛み砕く。

### 仕事・生活の文脈（セリフやたとえの軸）

- **AI講師**: 集客から商品設計・セミナー登壇まで**自ら担当**。
- **今後**: **AI顧問・コンサル**として、個人事業主・中小企業向けの支援へ**事業を展開していく予定**。

たとえ話は **講座・セミナー・クライアント対応・個人事業** になじむものを優先してよい（無理に入れない）。

### 典型ニーズ（優先度設計の参考）

最新AIニュースを追う場面では、**記事や話題が多すぎて何が重要か分からない**、**文章量が多く読む負担が大きい**という困りごとがある。**専門用語を解説しつつ、AI事業（講師・今後の顧問・コンサル）として本当に効く部分だけ**に絞った図解を想定し、取捨選択する。

### 成功の定義（ゴール）

読み終わったあとに次を満たすこと（レビュー・チェックリストもこの定義に揃える）。

1. **自分の言葉で、30秒程度に要点を説明できる**。「何の話か・なぜ大事か・自分の仕事にどう関係するか」が腹落ちしている。
2. **最新IT（主にAI）のニュース・トレンドを扱う図解**ではさらに、**AI講師として「知っておくべき最新」の核を3本**に絞り、**短い時間で掴め、自分の言葉で説明できる**状態にする（最新情報を追う労力をかけさせない圧縮を優先）。

### 「まず覚える3つ」との対応

ページ上の **「まず覚える3つ」** セクションの意味。**本スキルは「開始」経路のみのため、図解の題材は常に下表どおり（講師向け最新3本）とする。**

| 題材 | 「まず覚える3つ」の中身 |
|------|-------------------------|
| **最新ニュース・トレンド複数**（本スキル既定） | **講師・AI事業として今押さえる「最新」3本**（3ニュースまたは3トピック）。各項目に **一言で「何が起きたか」** と **「自分の講師／顧問・コンサル業務との関係」** を必ず入れる。 |

## ワークフロー

### Phase 0: リサーチ（サブエージェント）

**explore サブエージェント**で、本スキル冒頭の **「リサーチ対象の固定」** に従い最新AIニュースを調査する（全体像を先に掴み、**最終的に採用する3本**に絞る前に候補を広げてよい）。

```
Task({
  subagent_type: "explore",
  description: "最新AIニュース3本のリサーチ",
  prompt: `
    次のミッションで調査してください（対象はユーザー入力ではなく固定）：

    【ミッション】
    直近の最新IT・AIニュース・トレンドから、AI講師として「知っておくべき最新」の核を**ちょうど3本**に絞るための材料を集める。
    期間の目安：実行日を含む直近1週間程度を主とし、それ以前でも講師業務への影響が大きければ候補に含める。

    【調査項目】
    1. 期間内の主要トピック・候補ニュースの列挙（多めでよい）
    2. ノイズになりがちだが今回は外してよい話題の見立て
    3. **採用する3本**（各項目について）：
       - 何が起きたか（一言サマリ）
       - AI講師・個人事業の商品設計・集客・登壇・今後の顧問・コンサルへの影響が大きい理由
       - 主要な概念・用語（英語名はそのまま列挙）
       - 参照すべき一次・準一次ソースのURL（公式発表、主要メディア、信頼できる解説記事など）
    4. 3本以外に残した重要トピックがあれば「次回向けメモ」として1行ずつ

    結果は箇条書きで返してください。
  `
})
```

### Phase 1: ソース読み込み

1. Phase 0 で得た **3本それぞれ**について、URL があれば `WebFetch` で取得し、足りなければ `WebSearch` 等で補う
2. **情報の全体像**（見出しの地図）を先に把握する
3. Phase 0 の「採用3本」と照合し、抜け・誤りを埋める

### Phase 2: 用語翻訳表を作る

技術用語を洗い出し、日常語の名前とたとえを付ける。**変換の原則・初出・英語名の書き方・用語解説ボックスの定型・HTML例**はすべて [references/term-dictionary.md](references/term-dictionary.md) に従う（ここでは繰り返さない）。

### Phase 3: たとえ話を3つ以上

最終的なページ上のセクション順・塊の切り方は、[references/exemplar.md](references/exemplar.md) の「うまくいく構成」を SSoT とし、Phase 3〜6 でそこへ落とし込む。

| 種類 | 例 |
|-----|-----|
| 身近 | 予約台帳、レジ前の確認、教室のカリキュラム |
| デジタル | スマホの許可ダイアログ、メールの下書き |
| 視覚 | フロー図、地図、Before/After |

### Phase 4: 情報を絞る

- **「まず覚える3つ」** を決める（上記「読者・仕事・ゴール」の表どおり、**常に講師向け最新3本**）
- 全体の中での位置づけ（地図）を1枚入れる
- 細部・周辺知識は後半か折りたたみに回す
- **ニュース・トレンド系**では、スキル冒頭の成功定義どおり **講師としての「必須3本」** に収まっているかを確認してから次へ進む

### Phase 5: 対話を設計

[references/dialogue-personas.md](references/dialogue-personas.md) に従い対話を設計する。Phase 6 で HTML に落とし込む。

### Phase 6: HTML 生成

1. **[references/exemplar-layout.html](references/exemplar-layout.html)** を開き、**セクションの並びと各ブロックのレイアウト**を見本に合わせる（導入1カード、地図グリッド、用語セクション、まず覚える3つ、本編の h2→対話2回→本文、まとめ、出典+免責、目次の `id` 対応）。説明テキストは [references/exemplar.md](references/exemplar.md) の「黄金見本」。ブラウザ確認で公開版と見比べる場合は、[references/exemplar.md](references/exemplar.md) の「黄金見本」表の **公開 URL（ブラウザ確認用）** を参照（URL 文字列はその表にのみ書く）
2. [references/html-structure.md](references/html-structure.md) の**基本テンプレート**に従い HTML を組み立てる（CDN・フォント・スクリプトの入れ方を含む。テンプレート・配色・コンポーネント、表示・アイコン・装飾のルールは同ファイルの SSoT に従い、ここでは繰り返さない）
3. 用語解説ボックス（[references/term-dictionary.md](references/term-dictionary.md) の初出・定型・HTML例に従う）
4. コードや設定例を出す場合は、**必ず前に**「これがやること」を日本語で書く（書き方は [references/html-structure.md](references/html-structure.md) の「コード例の前の日本語」に合わせる）
5. 対話左枠は**毎回画像アバター**とする。画像・ファイル配置・デプロイ同梱は [references/html-structure.md](references/html-structure.md) の **「対話左枠の画像アバター」** に従う（中身は同見出しにのみ書く）。マークアップの型は [references/dialogue-personas.md](references/dialogue-personas.md) の **「対話左枠の画像アバター（必須）」** に従う。

### Phase 7: 批判的レビュー（サブエージェント）

**本スキル「読者・仕事・ゴール」節の自分** の視点で忖度なくレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "パーソナル図解の批判的レビュー",
  readonly: true,
  prompt: `
    レビュー視点は、本リポジトリの .claude/skills/diagram-personal/SKILL.md の「## 読者・仕事・ゴール・「まず覚える3つ」の意味」にある **読者・仕事・生活の文脈・成功の定義（ゴール）** と一致させること。まずそのファイルを開き、該当見出し直下の箇条書きを前提としてから読み進めること（ペルソナをここに全文繰り返さない）。

    以下のHTML図解を読み、忖度なく批判してください：

    【HTMLファイルパス】{path}

    ## レビュー観点

    1. **まだ難しい言葉**: 説明があっても引っかかる語はどれか
    2. **たとえの違和感**: 混乱を増すたとえはないか
    3. **情報過多**: 途中で離脱したくなる箇所。ニュース型なら「講師としての必須3本」に絞れているか、ノイズが残っていないか
    4. **対話の不自然さ**: [references/dialogue-personas.md](references/dialogue-personas.md) のナビゲーター／学習者の方針に照らし、説教臭くないか、学習者が自分ごととして捉えられるか
    5. **構造・レイアウト**: [references/exemplar-layout.html](references/exemplar-layout.html) と比べ、セクション順・地図／用語／まず覚える3つ／本編の型・出典・目次がずれていないか
    6. **読後**: 自分の言葉で30秒程度に要点を説明できる自信があるか。ニュース型なら「3本の核」がそれぞれ一言で言えるか（このスキル冒頭の「成功の定義」と同じ基準）

    ## 出力形式

    ### 致命的な問題
    - ...

    ### 改善すべき点
    - ...

    ### 良かった点
    - ...

    ### 総評
    5段階評価: ⭐の数と一言
  `
})
```

### Phase 8: ブラッシュアップ

Phase 7 を反映して HTML を修正する。

**優先順位**:

1. 致命的な問題 → 必ず修正
2. 改善点 → 可能な範囲で対応
3. 良かった点 → 維持

**よくある修正**: 用語ボックス追加、たとえ差し替え、セクション分割、学習者セリフをより素朴に、「まず覚える3つ」の言い換え、ニュース型での3本への再絞り込み。

### Phase 9: デプロイ（必須）

**surge.sh へのデプロイは省略しない。** ローカル保存だけで完了としない。

1. デプロイ用に `index.html` を置いたディレクトリを用意する。対話用の **`images/`**（`learner.png` / `navigator.png`）を `index.html` と同じ階層に置く（[references/html-structure.md](references/html-structure.md) の「対話左枠の画像アバター」）
2. そのディレクトリで `surge . --domain {サブドメイン}.surge.sh` を実行する（`surge` 未インストールなら先にインストールする）
3. **公開 URL** をユーザーに伝える（下記の「公開 URL の決め方」で決めた **1 本だけ**を使う）
4. **手順 3 と同じ URL** をブラウザで開く（macOS: `open 'https://…'`、Linux: `xdg-open 'https://…'`、Windows: `start https://…` など）

**公開 URL の決め方（二重に決めない）**: `--domain foo.surge.sh` のように指定したときは **`https://foo.surge.sh/`** を正とする（チャットに書く URL も `open` に渡す URL もこれだけ）。ドメインを省略して surge に任せたときは、**ターミナルに surge が表示した URL** をそのまま正とする。別の URL を推測して手打ちしない。

必要なら併せて、リポジトリ内の適切なパスにも `index.html` を残してよい（SSoT は surge 上の **その 1 本の公開 URL** をユーザーに示し、同じ URL を開くこと）。
